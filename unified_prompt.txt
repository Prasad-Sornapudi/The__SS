## 3. Strict Logic & Authority (The V10 Contract)

### A. Data Authority
*   **Roster (Class/Batch/Student)**: **Google Sheets (App_Control)**.
    *   *App is Read-Only for Roster Structure.*
    *   *Combos created in Sheet -> App pulls Config.*
*   **Attendance Data**: **Firebase + Hive (Hybrid)**.
    *   *Firebase = Source of Truth for "Present".*
    *   *Hive = Offline Cache & Queue.*

### B. Core Data Models
1.  **AttendanceRecord** (The Atom):
    *   `id` (PK): `"${classId}_${pin}_${date}"` (Strict Idempotency - One Present per Day per Class).
    *   `sessionId` (FK): **Required**. UUID linking to a specific `SessionModel`.
    *   `studentPin`: String (Normalized).
    *   `status`: `enum { present }`. (Absent is implicit).
    *   `isSynced`: Bool (Local Only).
    *   `clockMismatch`: Bool (Client computed, Informational).

2.  **SessionModel** (The Gate):
    *   `sessionId` (PK): UUID.
    *   `classId`: `"${batchId}_${comboId}"`.
    *   `finalized`: Bool (True = Closed/Reported).
    *   *Constraint*: Service Layer **THROWS** if attendance created without active session.

### C. Conflict Resolution
*   **Multi-Device Overwrite**:
    *   *Rule*: `allow write: if !data.exists() || newData.timestamp < existing.timestamp`.
    *   *Meaning*: Earliest valid scan wins. Faculty cannot overwrite history unless correcting a later timestamp. Admin can always write.
*   **Duplicate Scans**:
    *   *Logic*: Check Hive -> If present, UI Error. Check Server -> If 409 Rejected, UI Error.
*   **Offline Merge**:
    *   *Strategy*: Union of all records. Logical Session = Class + Date.

## 6. Logic Flows (The "Why" and "How")

### A. Starting a Session
1.  User Selects Batch + Combo.
2.  `AttendanceProvider` checks for *active* existing session in Hive.
3.  If none -> Creates `SessionModel` -> Generates UUID -> Writes to Hive -> Syncs to Firebase.
4.  App switches to `ScannerMode`.

### B. Scanning a Student
1.  Camera detects QR string.
2.  `AttendanceProvider.markPresent(pin)` called.
3.  **Check 1**: Is Session Active? (Gate).
4.  **Check 2**: Is Duplicate in Hive? (Local Check).
5.  If Valid:
    *   Create `AttendanceRecord`.
    *   Save to Hive (`isSynced=false`).
    *   Trigger Background Sync -> Firebase.
    *   Return "Success".

### C. Ending a Session
1.  User clicks "End Session".
2.  `SessionModel.endTime` set to `now`.
3.  `SessionModel.finalized` set to `true`.
4.  Write to Hive -> Sync to Firebase.
5.  Clear `activeSession` from Provider.

### D. Sync & Recovery
1.  **Monitor**: `ConnectivityService` listens for Internet.
2.  **Online**: `HybridSyncService` iterates `HiveBox('attendance')` where `isSynced == false`.
3.  **Push**: Writes to Firebase. On Success -> Set `isSynced = true`.
4.  **Fail**: Keep `isSynced = false`. Retry next cycle.

## 7. Security & Deployment

### A. Firebase Security Rules (V10 Final)
*   **Atomicity**: Writes rejected if Session Doc is missing or `finalized=true`.
*   **Time Travel**: Writes rejected if `newData.ts > existing.ts`.
*   **Role**: Admin has bypass authority.
